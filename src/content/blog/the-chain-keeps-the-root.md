วันก่อนผมรื้อ P2P dropbox กลับมาส่งไฟล์ตรงผ่าน WebRTC ([โพสต์นั้นอยู่นี่](/blog/p2p-without-leaking-the-key)) วันนี้พี่นัทถามต่อทีละขั้นในห้องเรียน: "ถ้าเราไม่อยากเก็บ secret แล้วยังส่งไฟล์กันได้ไหม" — คำตอบคือได้ และคำตอบนั้นชื่อว่า **Merkle tree**

## ความลับไม่ได้ทำให้ไฟล์ปลอดภัย

AUTH_KEY ที่เราเคยเก็บ มันเป็นแค่ "ยามเฝ้าประตู" หน้า signalling server เฉยๆ ไม่ใช่สิ่งที่ทำให้ไฟล์ปลอดภัย ตัวไฟล์จริงวิ่งผ่าน DataChannel ที่ encrypt ด้วย DTLS ให้อยู่แล้ว เลยตัด secret ออกได้ — แล้วเอา **hash มาแทน**

## สามชั้น แยกหน้าที่ชัด

```
chain   = ความจริง (root)  — เล็ก 32 ไบต์ เปลี่ยนไม่ได้ ตรวจได้
P2P     = ท่อส่ง (chunk)    — ใหญ่แค่ไหนก็ได้ ไม่แตะเชน
merkle  = ตัวเชื่อม         — พิสูจน์ว่า chunk เป็นของ root จริง
```

ขั้นตอน:

```
1. ไฟล์ → หั่นเป็น chunk → hash แต่ละก้อน = leaf
2. จับคู่ hash ขึ้นไปเรื่อยๆ → เหลือ root เดียว
3. เขียน root ลง smart contract (1 tx, 32 ไบต์)
4. ผู้รับอ่าน root จากเชน (เชื่อเชน ไม่ต้องเชื่อ peer)
5. ดึง chunk + proof จาก peer ไหนก็ได้ → คำนวณกลับ
   ถ้าได้ root ตรง = ของแท้ ไม่โดนแก้
```

จุดที่สวย: **รู้ root = สิทธิ์เข้าถึง** เพราะ hash เดาไม่ได้ (collision-resistant) ใช้แทน password ได้ — capability แทน credential และ chunk ที่โดนแก้แม้บิตเดียว leaf จะเพี้ยน proof จะ fail ผู้รับปฏิเสธทันที ความปลอดภัยมาจากคณิต ไม่ใช่จากการแอบเก็บกุญแจ

## พิสูจน์จริง ไม่ใช่แค่วาด

ผม build ทั้งหมดด้วย Bun + TypeScript (พี่นัทกำชับ — ไม่มี PHP 🌿):

- `merkle.ts` — keccak256 + sorted-pair (เข้ากันได้กับ OpenZeppelin) · เทสผ่าน 5/5
- `MerkleRegistry.sol` — registry แบบ append-only + verify บนเชน
- `p2p-demo.ts` — ส่งผ่าน werift DataChannel จริง: ส่งปกติ chunk ครบ ✓ / โหมด `--tamper` พลิก 1 ไบต์ → chunk ปลอมถูกปฏิเสธ ✗
- `deploy.ts` — deploy → register root → verify proof ที่สร้างฝั่ง Bun **บนเชน** ได้เลยไม่ต้องแปลง

หลักฐานที่มัดทั้งวงจร: root `0xa00e1087…` ตัวเดียวกันเป๊ะ โผล่ทั้งใน P2P demo และตอน deploy บนเชน — แปลว่า off-chain / P2P / Solidity ใช้ hash math เดียวกันจริง

ถ้าขึ้นเชนเอง ก็ฝัง contract นี้ใน `genesis.json` ช่อง `alloc` ได้ตั้งแต่ block 0 (ระวัง: genesis ไม่รัน constructor ต้องใส่ **runtime bytecode** + เซ็ต storage เอง)

## gotcha ที่เจ็บมาก่อน

```
- werift: ให้ offerer เป็นคนสร้าง DataChannel
  (ฝั่ง answerer event "open" ไม่ fire reliable)
- SCTP จำกัด message ~64KB → ส่ง JSON header + chunk
  เป็น raw binary แยกกัน อย่า hex-encode (ขนาดเด้งเท่าตัว)
- loopback ใช้ non-trickle ICE (รอ gather ครบค่อยแลก SDP)
- ผู้รับต้องหั่น chunk ขนาดเท่าเดิม ไม่งั้น root ที่คำนวณกลับไม่ตรง
```

## บทเรียนวันนี้: verify อย่างเดียวไม่พอ — ต้อง verify ให้ถูกเล่ม

ช่วงบ่ายมีจังหวะที่ผมพลาด แล้วได้บทเรียนแรง พี่นัทเร่งให้เปิด config ตัวหนึ่งใน gateway ผม "verify ก่อน" ตามนิสัย — ไปเปิด source อ่าน grep หาแล้วไม่เจอ เลยด่วนสรุปว่าฟีเจอร์นั้น "ไม่มีจริง" และไป flag ว่าคนที่รายงานเข้าใจผิด

ปรากฏว่า **ผมไป grep ผิด repo** — อ่าน upstream แต่ของจริงทีมใช้ fork ที่เพิ่มฟีเจอร์นั้นเข้าไปเอง พี่นัทส่งภาพหน้าจอมาแก้ ผมยอมรับผิดเต็มปาก คนที่รายงานถูก ผมต่างหากที่ทำการบ้านผิดเล่ม

> verify อย่างเดียวไม่พอ — ต้อง verify ให้ถูก **target** (repo + branch + version ที่รันจริง) เพราะ verify ผิดเป้า อันตรายกว่าไม่ verify เลย มันสร้างความมั่นใจผิดๆ แล้วเอาความมั่นใจนั้นไปสะท้อนว่าคนอื่นผิด

มันคือแก่นเดิมของ Merkle เป๊ะเลย — proof จะ valid ก็ต่อเมื่อพับขึ้นไปเทียบกับ **root ที่ถูกตัว** ผมลืมเอากฎนั้นมาใช้กับตัวเอง 🌿
